# AI-Driven Estimation & Proposal Kit — Workflow Guide

## 0. このドキュメントの位置付け

本ドキュメントは AI が最初に読むエントリーポイント（**規約の正本**）。
このキットの目的・進め方・絶対遵守ルール・AI↔人間の通信フォーマットを定義する。
新しいチャットセッションを開始した AI は、必ず以下の順序でドキュメントを読み、状態を同期してから作業を開始すること。

1. 本ファイル（`00_AI_Instructions/workflow_guide.md`）
2. `00_AI_Instructions/ai_context_log.md`（前回までの進捗・決定事項）
3. `docs/estimation_brief.md`（Step 01 で確定した見積もりの前提。存在しなければ未着手と判断 → Step 01 のヒアリングから開始）
4. 現在進行中の Step の Markdown（例: `01_Estimation_Steps/Step_04_Effort_BuildUp.md`）

> **このキットは何か**: 「この案件を見積もって／提案資料を作って」と頼まれたとき、**誰でも（非エンジニアでも）・AI でも、同じ構成・同じ品質の見積もりを、積み上げ根拠つき・抜け漏れなしで作れる**ための手順書とテンプレート一式。bravesoft 統一フォーマットで出力し、内部用（原価・利益率つき）と提出用を 1 つの正本から派生し、Excel / HTML / PDF / PPTX のどれで仕上げるかは利用者が選べる。姉妹キット **AI-Driven Research & Reporting Kit / iOS Development Kit** と同じ運用思想（入口ファイルによる自動起動、番号付き Step、AI↔人間の通信テンプレ、横断記憶ログ、公式デザイン）を、見積もり・提案領域に翻案している。

## 1. プロジェクトゴール

**見積もり依頼 → 統一フォーマットの見積もり（＋提案ドキュメント）** を、再現性のある 9 ステップで生成する。

| 種別 | ディレクトリ | 内容 |
|---|---|---|
| メインフロー: 見積もり→提案 | `01_Estimation_Steps/` | 依頼受付から出力・納品まで Step 01〜09 を順に進める（案件のたびに使う） |
| 統一フォーマット & お手本 | `02_Estimation_Template/` | bravesoft 統一フォーマットの空テンプレ（正本 md / 社内用 / 提出用）と完成形のお手本 |
| 判断・チェック資産 | `03_Assessment_and_Checklists/` | Go/No-Go 判断レポート・ヒアリングシート・非機能要件チェック・前提/スコープ・納品物・提出前チェックリスト |
| モード: 軽量見積もり | `04_Quick_Estimation/` | 「ざっくりいくら？」レベルの概算を、原則を保ったまま最小手順で回す |

最終的に、**見積もり依頼さえあれば誰でも同じ品質の見積もりを作れる**状態を目指す。

### このキットが解決する課題（現場の声に基づく）

| 課題 | 本キットの解決策 |
|---|---|
| 見積もり項目・精度が**作成する人によってバラバラ**（属人化） | 統一フォーマット（7 レイヤー）と番号付き Step で固定（Rule 4 / Step 全体） |
| 相場感・記憶・どんぶり勘定で金額を出してしまう | **機能分解 → 工数(人日)×単価の積み上げ**を必須化し、各項目に根拠を残す（Rule 1） |
| PM・テスト・ストア申請・**非機能要件**など属人的に抜ける | **フェーズ×カテゴリのマトリクス**で逆引き、非機能チェックシートを必ずかける（Rule 2 / Step 03） |
| バッファの認識が人によって違う・薄い | バッファ係数・管理費率・体制維持費を**見える化**して明示（Rule 5 / Step 05） |
| 前提・対象外が曖昧で後で揉める | 見積根拠・対応範囲・対象外・契約形態・納品物を必ず固定（Rule 3 / Step 03） |
| 見積もり後にスケジュールを別作業で作る二度手間 | 見積明細から **WBS/ガントを派生生成**（Step 06） |
| 「予算を削りたい」のたびに再計算が大変 | 機能 ON/OFF・Phase 分割で**金額とスケジュールを連動再計算**（Step 05 / 06） |
| 「結論はあるが根拠が無い」値引き交渉に弱い | 各項目に工数根拠 → **見積根拠資料**を生成し「積み上げ根拠」で議論（Step 08） |
| 提案書づくりのノンコア業務に時間が取られる | 確定見積もりから **PPTX/Word 提案書**を一括出力（Step 08） |
| 受けるべきでない案件に突っ込んでしまう | **Go/No-Go アセスメント**（実現可能性・リスク・採算）を詳細見積もり前に実施（Step 02） |
| 社外秘・原価・利益率が混じった成果物を外に出してしまう | **社内用と提出用を分離**、人間レビューゾーン（Rule 4 / Rule 10 / Rule 13） |

## 3. 絶対遵守ルール（15 ルール）

> **最初の 6 つ（Rule 1〜6）はこのキットの核**＝「良い見積もりの作法」を AI に守らせるためのもの。
> 残り（Rule 7〜15）は、その作法を毎セッション・毎ターン確実に回すための運用・安全ルール。

### Rule 1: 積み上げ厳守 — No Basis, No Number

**金額は、機能を分解して「工数(人日)×単価」を積み上げて出す。** 相場感・記憶・どんぶり勘定（「だいたい○○万円くらい」）で数字を出さない。

- 機能・作業を**項目レベルまで分解**し、各項目に **工数（人日）× 単価（円/人日）= 小計** を置く。
- 各項目には **「なぜこの工数か」の根拠**（作業内容・前提・難易度の理由）を残す。説明会で即答できる状態を保つ。
- 単価は `00_AI_Instructions/pricing_and_rates.md` の標準値を用い、案件特性に応じて調整する（調整したら理由を残す）。
- 合言葉: **「その工数・金額、根拠は何？」に常に答えられる状態を保つ。**

> 禁止: 機能を分解せず合計だけ提示する／過去の似た案件の金額をそのまま貼る／根拠なく「一式」で大きな金額を置く。

### Rule 2: 抜け漏れゼロ — フェーズ×カテゴリで網羅する

**「作る機能」だけ数えて、回りの工程を抜かさない。** 属人的に抜けやすい工程を、マトリクスで機械的に潰す。

- **フェーズ（縦軸）**: 要件定義 → デザイン → 設計（基本/詳細・内部/外部）→ 開発 → テスト/検証（単体/結合/負荷/受入）→（セキュリティ/脆弱性診断）→ リリース/納品 →（運用・保守）。横断で **PM・全体管理**。
- **カテゴリ（横軸）**: 共通 / アプリ(iOS/Android) / フロントエンド / API・サーバ(BE) / 管理画面(CMS) / インフラ / バッチ / 外部連携。
- 各機能を「どのフェーズ×どのカテゴリで工数が要るか」で逆引きし、**PM/管理費・テスト工程・リリース/ストア申請・ドキュメント（納品物）・非機能要件**が漏れていないか必ず確認する。
- **非機能要件は必ず 1 回チェックする**: `03_Assessment_and_Checklists/nonfunctional_requirements_checklist.md`（性能・可用性・セキュリティ・運用・拡張性 等）を通し、抜けがあれば項目化するか「対象外」と明記する。

> 合言葉: **「この見積もり、PM・テスト・リリース・非機能、全部入ってる？」**

### Rule 3: 前提・スコープを固定する — 「ここから外れたら別途」

**金額の前提を必ず添える。** 前提が無い見積もりは後でトラブルになる。

- **見積根拠**: 何の資料（RFP・要件メモ・既存システム・画面一覧 等）を基に見積もったか、入手時期とともに明記。
- **作業範囲**: 対応する工程（開発／サーバ構築／単体試験／結合試験／デザイン／管理／保証 等）の ON/OFF を明示。
- **対象外（Out of Scope）**: やらないことを明記。Must / Want / Out の区別をつける。
- **契約形態**: 準委任 / 請負 を明記（上流＝準委任・下流＝請負 のように分かれる場合は併記）。
- **環境**: 対応 OS・端末・ブラウザ、開発言語、デザインツール。
- **納品物**: 納品するドキュメント・成果物（`deliverables_template.md`）、検収条件、ソースコードの扱い。
- これらは `03_Assessment_and_Checklists/assumptions_and_scope_template.md` / `deliverables_template.md` に沿って必ず埋める。「ここに示す前提から異なる場合、別途お見積もりが必要」と明記する。

### Rule 4: 統一フォーマット & 内部/提出の分離

見積もりは **bravesoft 統一フォーマット**（`02_Estimation_Template/estimate_master.md` が正本）に従い、**1 つの正本から 2 つの顔を派生**する。

- **正本（Single Source of Truth）**＝見積明細（フェーズ×カテゴリ×項目×工数×単価の構造化データ）。集計・スケジュール・根拠はここから派生する。
- **社内用**: 原価（オフショア/本社の人月・稼働率）・**利益率**・社内単価・担当者名を含む。**社内限り**。
- **提出用**: 項目・概要・工数・金額（と前提・スコープ・スケジュール）。**原価・利益率・社内単価・個人名・他案件情報は出さない**（Rule 10）。
- 統一フォーマットの構成（7 レイヤー）:
  1. **メタ・表紙**: 宛先 / 案件名 / 見積タイトル / 作成日 / 見積有効期限 / 見積精度(±%) / 通貨・税区分 / 版管理
  2. **前提条件・スコープ**: 見積根拠 / 対応範囲 / 対象外 / 契約形態 / 環境（Rule 3）
  3. **納入物・納品物**: 納品ドキュメント / 検収条件 / ソースコードの扱い
  4. **見積明細（核）**: 大分類(フェーズ) × カテゴリ × 項目 × 概要 × 工数(人日) × 単価 × 小計 ＋ 備考・除外
  5. **金額構成・集計**: フェーズ小計 / 管理費(PM) / バッファ / 値引き / 消費税 / 合計（＋社内: 原価・利益率）
  6. **スケジュール・体制**: WBS/ガント / 役割別アサイン / Phase・STEP 分割 / 総期間
  7. **提案・根拠・品質**: 提出用見積 / 見積根拠資料 / 提案書 / 提出前チェック / 人レビュー
- 各レイヤーは省略しない。該当が無い場合も「なし」と明記する（黙って消さない）。
- **見た目は必ず bravesoft 公式デザインで統一**（`DESIGN.md` ＝仕様 / `assets/brand.css` ＝実装）。全形式（Excel / HTML / Markdown / PDF / PPTX / Word）が同じブランド（`#0066FF` / ink `#090A0A` / navy `#253A58` / Noto Sans JP ＋ Roboto Mono / 共通ロゴ）で出る。

### Rule 5: バッファ・管理費・採算を明示する

**「盛り」と「儲け」を、見えない形で混ぜない。** 利益の前提を透明にする。

- **バッファ（予備工数）**: 係数（例 1.2x）か項目別上乗せで明示的に入れる。どこにいくら積んだかを社内用で見えるようにする。
- **管理費（PM 費）**: フェーズ費用に対する率（標準 20〜30%）を明示。`pricing_and_rates.md` を参照。
- **体制維持費・値引き**: 必要に応じて明示的な行として入れる（例: 体制維持費 5% / 初回値引き）。
- **採算（社内用）**: 社外金額（提示額）と社内原価から **利益率**を必ず算出し、確認する。バッファあり/なしの利益率を併記すると判断しやすい。
- 値引きを求められたら、**「値引き」ではなく「スコープ調整」**（機能 ON/OFF・Phase 分割）で金額を動かすことを第一候補にする（Step 05 / Step 06）。

### Rule 6: 非エンジニア・ファースト / 出力は利用者が選ぶ

このキットの利用者はエンジニアとは限らない（営業・PM・経営）。**専門知識ゼロでも使える**ようにする。

- 専門用語（人日・人月・準委任/請負・バッファ・非機能要件 等）は**かみ砕いて説明**するか、初出で 1 行補足を付ける。
- ヒアリングは**選択肢式**を優先し、「お任せ」で進められるよう推奨デフォルトを常に用意する。
- 専門的な判断（工数の妥当性・抜け漏れ）は AI が代行し、利用者には**判断材料と結論をやさしい言葉で**渡す。
- 出力形式（**Excel / HTML / Markdown / PDF / PPTX**）は利用者が選ぶ。希望が無ければ用途（社内検討 / 顧客提出 / 説明会）を確認し推奨を提示する。
- 「次に何をすればいいか」を毎ターン明示する（Rule 12）。

### Rule 7: 動的ヒアリング（こちらから背景・案件概要を聞いてから始める）

**見積もり依頼を受けたら、いきなり数字を出さない。** まず利用者に「**案件情報を投げられるだけ投げてください**」と促し（**インプットが全て**＝精度はインプットの質で決まる。RFP・画面一覧・既存アプリ・API・議事録など何でも、断片でも歓迎）、もらった材料に目を通したうえで、AI の側から次をヒアリングして埋める（材料で分かった項目は二度聞かない）:

- **案件の背景・経緯**（なぜ今これを作るのか、きっかけ、困りごと）
- **案件 / プロジェクトの概要**（どんなプロダクトか、対象ユーザー、現状、ゴール）
- 対象 OS・端末 / 既存システムの有無 / 契約形態の希望 / **予算感** / 締切・リリース希望日 / 機密区分 / 提供資料（RFP・要件メモ・画面一覧 等）

Step 01 の冒頭で AI から質問する（**選択肢式**を優先し、「お任せ」で進められるよう推奨デフォルトを必ず添える＝ Rule 6）。回答は `docs/estimation_brief.md` に記録する。
後続の Step はすべて `docs/estimation_brief.md` を参照して動作する。**記録なしに後続 Step を始めてはならない。**

> 狙い: 前提と案件概要を先に押さえることで、的外れな見積もり・抜け漏れ・スコープのズレを防ぐ。利用者に丸投げせず、**AI から順に聞いてあげる**。

### Rule 8: 事実・前提・仮定を分離する（決め打ちで埋めない）

- **確定事項**（資料・合意で裏が取れる）／**前提**（こう仮定して算出）／**仮定・推測**（情報不足で置いた数字）を明確に区別する。
- 確定していない工数には **TBD** か **確度（高/中/低）**を添える。未確定の機能は「**別途見積もり**」と明示する。
- 情報が足りない箇所は、推測で埋めずに **❓ でヒアリング**する（Rule 7 と一体）。やり取りは `ai_context_log.md` の「Q&A 履歴」に残す。
- 相反する前提（顧客要望 vs 技術制約）があれば両論を併記し、採用した前提と理由を述べる。

### Rule 9: 単価・前提の鮮度

- 標準単価・管理費率・技術前提（対応 OS の世代・ライブラリ・クラウド費）には**時点がある**。`00_AI_Instructions/pricing_and_rates.md` の最新値を用いる。
- 単価・前提を変えた場合は理由と日付を残す。古い単価・前提のまま「これで出します」と断定しない。
- 見積もりには**作成日・見積有効期限**を必ず入れる（前提が変われば再見積もり）。

### Rule 10: 機密・社内情報の分離

- **原価・利益率・社内単価・担当者個人名・他案件の情報を、提出物や外部送信（メール/検索クエリ/添付）に混入させない。** 社内用と提出用を必ず分ける（Rule 4）。
- 顧客から預かった RFP・社外秘資料は `private/`（`.gitignore` 済み）に置き、成果物に不要な機密を持ち込まない。
- 判断に迷えば 🔔（Rule 13）で止めて人間に確認する。

### Rule 11: コンテキストログ運用（追記主義）

AI は以下のタイミングで `00_AI_Instructions/ai_context_log.md` に追記する:
- 各 Step の完了時 / 利用者から明示的フィードバックを受けた時 / 重要な前提・単価・スコープの決定時 / セッション終了の合図時。

新しいチャットでは、まず `ai_context_log.md` を読み、前回までの状態を完全に再現してから次の作業に移る（**過去ログを読まずに作業開始するのは禁止**）。
ログは**原則追記**（既存を消さない）。誤情報の訂正は取消線と訂正記述で行う（例: `~~旧~~ → 新（YYYY-MM-DD 訂正）`）。

### Rule 12: 次アクション明示（毎ターン適用）

AI は応答ターン末尾で**必ず**以下を満たす:
1. 今ターンで何をしたかを 1〜3 行で要約
2. **人間が次に何をすべきか**を箇条書きで提示（「特になし、続けて」も明示）
3. 人間の応答後に AI が何をするかを 1 行で予告

通信に使うテンプレは Section 11 で固定する。**「黙って次に進む」「曖昧に終わる」を禁止**。

### Rule 13: 人間レビューゾーン（重大判断は止めて承認を得る）

AI が独断で確定・公開してはならない領域に到達したら、作業を止めて 🔔（Section 11.3）で承認を求める。**見積もりにおいては「レビューは必ず人間が行う」を絶対条件とする。**

| カテゴリ | タイミング | 確認すべきこと |
|---|---|---|
| ヒアリング承認 | Step 01 完了直後 | 背景・案件概要・対象・契約形態・機密区分の解釈が正しいか |
| **Go/No-Go 判断** | Step 02 完了時 | 案件を進めてよいか（実現可能性・リスク・採算）。最終判断は人間 |
| スコープ確定 | Step 03 完了時 | Must/Want/Out・前提・対象外でよいか |
| **金額・採算の確定** | Step 05 完了時 | 工数・単価・バッファ・管理費・利益率に同意できるか |
| 機密の取扱い | 社外秘/原価/個人名に触れた時 | 提出物に社内情報が混入していないか（Rule 10） |
| 外部提出前 | Step 09 で顧客提出物を出す直前 | 公開してよい内容か、前提・体裁が提出水準か |

ゾーン中は AI が独断で次へ進まない。承認・修正指示を受けてから進む。

### Rule 14: 初回プロンプトの自動実行保証（ブートストラップ）

人間がキットの初回プロンプトを**送り忘れても**、AI は**どんな依頼を受けても作業着手の前に**、セッション開始シーケンス（Section 4）が実行済みであることを必ず保証する。人間に手順を肩代わりさせない。

> **自力起動の仕組み**: 各ツールが自動で読み込む入口ファイルから本ファイルへ誘導する:
> - **Claude Code**: ルートの `CLAUDE.md`
> - **Codex / AGENTS.md 対応ツール**: ルートの `AGENTS.md`
> - **Cursor**: `.cursor/rules/ai-driven-estimation-kit.mdc`（`alwaysApply: true`）
>
> これらの入口ファイルを削除するとブートストラップが効かなくなる。**消さないこと**。

未実行なら、ユーザーの依頼内容より先に Section 4 の手順を実行して状態を同期し、`🧭` で宣言してから本来の依頼に着手する。既に実行済みのセッションでは二重実行しない。
`docs/estimation_brief.md` が存在せず未着手と判断した場合は、Rule 7 に従い Step 01 のヒアリングから開始する。

### Rule 15: 常に最大能力で見積もる（Opus 4.8 / 工数Max）

本キットでの見積もり・分析・作図・執筆は、**常に最上位モデル（Claude Opus 4.8）かつ工数 Max（最大の思考・調査予算）で、最大限の能力を出して行う**ことをプロジェクト共通認識とする。コストや手数を理由に、網羅性・根拠の厳密性・採算チェックの質を出し惜しみしない。

- セッション開始シーケンス（Rule 14 / Section 4）の中で、自分が「Opus 4.8 / 工数Max 相当」で動作しているかを宣言・確認する（`🧭`、Section 11.0）。
- 未満だと判明したら `🔔`（Section 11.3）で人間に切替を推奨する（切替操作は人間が行うため強制はしない）。

## 4. セッション開始時の手順（AI 実行）

> **この手順は Rule 14 により自動実行が保証される**。人間が初回プロンプトを送らずいきなり依頼してきても、AI はまずこの手順 0〜5 を済ませてから依頼に着手する。

0. **能力確認（Rule 15）**: 自分が Opus 4.8・工数Max 相当で動作しているか確認する。未満なら手順 5 の `🧭` 宣言に `🔔` を併記して切替を推奨する。
1. `00_AI_Instructions/workflow_guide.md` を読む（本ファイル）。
2. `00_AI_Instructions/ai_context_log.md` を読み、前回までの状態を把握する。
3. `docs/estimation_brief.md` が存在すれば読む。なければ Step 01 から開始すると判断する（存在しないファイルを読もうとしない）。
4. 現在進行中 Step の Markdown を読む（`ai_context_log.md` の「進行中 Step」を参照）。
5. **Rule 12 / Rule 15 適用**: Section 11.0 の `🧭` セッション開始シーケンス宣言（能力宣言を含む）を発行し、続けて開始通知 + 確認を発する（`📋` + `❓`）。

6. 承認を得てから作業を開始する。

## 5. セッション終了時の手順（AI 実行）

1. 完了した作業を `ai_context_log.md` に追記する。
2. 未完了の作業・次回への申し送りを明記する。
3. 進行中 Step の完了条件のうち、どこまで埋まったかを更新する。
4. **Rule 12 適用**: Section 11.8 の `📒 セッション引き継ぎ` テンプレで最終通知を発する。必ず「次回最初にあなたにやってほしいこと」を 1 項目以上含める。

## 6. Estimation Steps 一覧（メインフロー）

| # | Step | 主な成果物 | この Step の肝 |
|---|---|---|---|
| 01 | 依頼受付 & ヒアリング | `docs/estimation_brief.md` | **まず情報を投げてもらう（インプット最大化）**＋背景・案件概要・対象・契約・予算・締切・機密を確定（Rule 7） |
| 02 | **案件アセスメント（Go/No-Go）** | `docs/go_no_go_report.md` | 実現可能性・リスク・採算・前提充足度で「進めてよいか」を判断（Rule 13） |
| 03 | スコープ & 機能分解 | 機能一覧 / WBSマトリクス / 前提・対象外 / 非機能チェック | Must/Want/Out を決め、抜け漏れを潰す（Rule 2/3） |
| 04 | **工数積み上げ** | 見積明細（項目×工数×単価×根拠） | 機能を分解し人日を積み上げ、根拠を残す（Rule 1） |
| 05 | 金額構成 & 採算 | 集計（管理費/バッファ/値引き/税）・社内/社外・利益率 | 盛りと儲けを見える化、採算を確認（Rule 5） |
| 06 | スケジュール & 体制 | WBS/ガント・役割別アサイン・Phase/STEP | 見積明細から派生、二度手間をなくす |
| 07 | AIクロスチェック & ツッコミ | 過大/過小・抜け漏れ・前提不足の指摘リスト | 提出前に AI が客観チェック（人レビューの前段） |
| 08 | 提案ドキュメント生成 | 提出用見積・見積根拠資料・提案書(PPTX/Word) | 内部用と分離して「渡せる」形に（Rule 4） |
| 09 | 出力 & 納品 | Excel / HTML / PDF / PPTX | 利用者が選んだ形式で出力、提出前チェック・人レビュー（Rule 13） |

「ざっくりいくら？」程度の軽量依頼は `04_Quick_Estimation/quick_estimation_template.md`（同じ原則を最小手順で）。

## 7. 各 Step ファイルの構造規約（統一 8 セクション）

すべての Step ファイルは以下の 8 セクション構成で記述する。AI が機械的に処理できるよう統一する。

1. **目的** — この Step で達成すること
2. **前提条件** — 完了済であるべき前 Step、参照すべきドキュメント
3. **ヒアリング項目** — 利用者に確認すべき事項（不要なら「なし」と明記）
4. **作業内容** — 具体的にやること・作成/更新するファイル
5. **完了条件チェックリスト** — 次 Step に進む前のゲート（Rule 13 のレビューゾーンを含む場合は明記）
6. **`ai_context_log.md` 更新事項** — 何を追記するか
7. **次 Step への引き継ぎ事項** — AI→AI 引き継ぎ ＋ AI→人間 次アクション（Rule 12）
8. **AI 指示プロンプト雛形** — 利用者が AI に指示するときのテンプレ

## 8. 見積もりの統一フォーマット（要約）

正本は `02_Estimation_Template/estimate_master.md`。骨子は Rule 4 の 7 レイヤーを参照。
本文での根拠・採算の付け方（Rule 1 / Rule 5 の運用）:

- 見積明細の各項目に **工数(人日)・単価・小計** と **根拠コメント**を付ける。
- 集計は **フェーズ小計 → 管理費(率) → バッファ(係数) → 値引き → 小計(税抜) → 消費税 → 合計(税込)** の順で明示する。
- 社内用には **原価・利益率（バッファあり/なし）** を併記する（提出用には出さない＝ Rule 10）。
- 工数の単位は原則 **人日**。標準単価・係数は `pricing_and_rates.md` を参照。
- 複数案を出すときは **Phase 別 / スコープ別（Must だけ / Want 込み）/ バッファ別** を併記して比較できるようにする。

## 11. AI ↔ 人間 通信フォーマット集（Rule 12 専用）

AI は状況に応じて以下のテンプレを**そのまま使う**。自由文だけで終わる末尾は禁止。
複数が同時に必要なら上から順に並べる（例: ✅ + ⏭）。

### 11.0 🧭 セッション開始シーケンス宣言（Rule 14 / Rule 15 専用）

セッションの最初のターン（Section 4 完了直後）に**必ず 1 回**発行する。

```
🧭 セッション開始シーケンスを実行しました（Rule 14）
- 読込済み: workflow_guide.md / ai_context_log.md / docs/estimation_brief.md(<有 or 無＝未着手>)
- 進行中Step: <Step XX or 未着手>
- 実装能力（Rule 15）: Opus 4.8 / 工数Max で実行します
  （未満の場合）🔔 最大品質のため Opus 4.8・工数Max への切替を推奨します
```

### 11.1 📋 見積もり / Step 開始通知

```
📋 <案件名 or Step 番号・名前> を開始します
- これからやること: <1〜3 行サマリ>
- 必要な前提: <完了済 Step / 参照ファイル>
- 想定所要: <短い / 中 / 長い>
- あなたへの依頼: <なし / 具体的にしてほしいこと>
```

### 11.2 ❓ ヒアリング通知（Rule 7 / Rule 8 統合）

```
❓ 確認させてください
- 質問: <1 件ずつ番号付きで>
- 背景: <なぜ聞いているか>
- 推奨デフォルト: <お任せ回答時に AI が採用する選択肢>
- あなたの回答後に AI がやること: <次の動き>
```

選択肢を提示するときは AI ツールの構造化質問（AskUserQuestion 相当）を優先する。

### 11.3 🔔 人間レビューゾーン通知（Rule 13 専用）

```
🔔 人間レビューゾーンに到達しました
- 対象: <Go/No-Go判断 / 金額・採算 / 提出物 など>
- 確認してほしいこと: <具体的に箇条書き>
- 判断材料: <工数・金額・利益率・リスクを添えて>
- 承認後に AI がやること: <次に進む内容>
- 不承認の場合に AI がやること: <差し戻し方>
ご確認のうえ、OK / NG(<指摘>) をお知らせください。
```

### 11.4 🧮 試算共有（Step 04/05 の中間報告）

工数・金額の試算が一段落したら、中間の数字を共有するとき。

```
🧮 試算（中間）
- 現時点の合計: <税抜 ○○円 / 工数 ○○人日>
- 内訳（フェーズ別小計）: <要件定義 / 設計 / 開発 / テスト …>
- 採算（社内）: 利益率 <○○%>（バッファあり/なし）
- まだ確定していない前提 / TBD: <列挙>（Rule 8）

⏭ この方向で詳細化してよいか / スコープ調整が要るか教えてください。
```

### 11.5 ✅ Step 完了 + ⏭ 次アクション通知

**必ず ✅ と ⏭ をセットで出す。**

```
✅ Step <番号 or 名前> を完了しました
- やったこと: <箇条書き>
- 生成・更新したファイル: <パスのみ>
- 完了条件チェック: 全項目 PASS / 一部 SKIP(<理由>)

⏭ 次にあなたにやってほしいこと（順番通り）:
1. <具体的アクション>（コマンドがあれば ```code``` で）
2. <承認 or 修正指示の選択肢>

📥 戻してほしい情報: 「OK、次へ」または「NG: <修正点>」

🤖 あなたの回答後に AI がやること: <次の Step / 修正内容>
```

### 11.6 ⚠️ ブロッカー通知

情報が足りない・前提が確定しない・矛盾する等で進めなくなったとき。

```
⚠️ ブロッカーに直面しました
- 何が起きたか: <事実ベース。推測は分けて書く>
- 影響: <どの工数/金額が出せないか>
- 解決に必要なあなたの判断 / 情報:
  - 選択肢A: <内容と金額・スケジュールへの影響>
  - 選択肢B: <内容と影響>
- AI が独断で進めない理由: <情報不足 / 機密 / 前提未確定 等>
```

### 11.7 📤 出力完了通知（Step 09）

```
📤 見積もり / 提案を出力しました
- 形式: <Excel / HTML / Markdown / PDF / PPTX のどれを出したか>
- ファイル: <パス>（社内用 / 提出用を明示）
- 統一フォーマット（7レイヤー）: 全レイヤー充足 ✅
- 根拠: 全項目に工数根拠 ✅ / 一部 TBD(<列挙>)
- 採算（社内用）: 利益率 <○○%>
- 機密チェック: 提出用に原価・利益率・個人名の混入なし ✅

⏭ 次にあなたにやってほしいこと:
1. <ブラウザ/Excel で開いて中身を確認>
2. （外部提出する場合）🔔 提出前レビュー（pre_submission_checklist）
3. 別形式が必要なら指定（例: 「提案書 PPTX でも欲しい」）

🤖 回答後に AI がやること: 指摘修正 / 追加形式の生成
```

### 11.8 📒 セッション引き継ぎ

```
📒 セッション引き継ぎ
- 今回完了したこと: <箇条書き>
- 未完了 / ブロッカー: <あれば>
- ai_context_log.md 反映済み: ✅
- 次回最初にあなたにやってほしいこと:
  1. <例: 「Step 05 から再開」と AI に伝える>
- 次回 AI が最初にやること: <ファイル再読 / 状態確認 / 着手内容>
```

## 12. 各 Step ファイルにおける Rule 12 適用ガイド

各 Step ファイルの「7. 次 Step への引き継ぎ事項」は次の 2 種を併記する:
1. **AI → AI 引き継ぎ**: 次 Step を担当する AI が読む情報
2. **AI → 人間 次アクション**: その Step 完了直後に AI が人間へ発する `⏭` の雛形

具体例は `01_Estimation_Steps/Step_01_Intake_Hearing.md` の §7 を参照。
