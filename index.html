AI-Driven Estimation & Proposal Kit

「この案件を見積もって」と頼めば、誰でも・AIでも、同じ構成・同じ品質の見積もりを、積み上げ根拠つき・抜け漏れなしで作れるキット。

これは何？

見積もり・提案づくりを AI（Claude Code / Cursor / Codex 等）と組んで標準化するためのドキュメント集です。キットを案件用フォルダに置いて AI を起動し、「○○の案件を見積もって」と頼むだけ。AI が ヒアリング → 案件アセスメント(Go/No-Go) → 機能分解 → 工数積み上げ → 金額・採算 → スケジュール → クロスチェック → 提案書 → 出力 の順に進め、bravesoft 統一フォーマットの見積もりを仕上げます。

社内に散在していた見積もりスタイル（BrSE 旧テンプレ／営業の概算／個人の独自テンプレ）を分析し、全社共通で必要な構成を 7 レイヤーに整理。それを「誰でも・AI でも同じ品質で」生み出す仕組みにしました。姉妹キット Research & Reporting / iOS Development Kit と同じ運用思想（入口ファイルによる自動起動・番号付き Step・AI↔人間の通信テンプレ・横断記憶ログ）を、見積もり・提案領域に翻案しています。

6つの約束（このキットの核）

良い見積もりの作法を、AIが必ず守るルール（Rule 1〜6）として埋め込みました。

🧮
1. 積み上げ厳守（No Basis, No Number）
機能を分解し 工数(人日)×単価 を積み上げ、各項目に根拠を残す。相場・記憶・どんぶり勘定で出さない。
🧩
2. 抜け漏れゼロ（網羅）
フェーズ×カテゴリで逆引き。PM・テスト・リリース/ストア申請・非機能要件を必ず通す。
📐
3. 前提・スコープを固定
見積根拠／対応範囲／対象外(Out of Scope)／契約形態／環境／納品物を明記。「外れたら別途」。
🔐
4. 統一フォーマット＆内部/提出の分離
1つの正本から、社内用（原価・利益率）と提出用を派生。社内情報を提出物に混ぜない。
💴
5. バッファ・管理費・採算を明示
バッファ・管理費・値引きを見える化し、利益率を必ず確認。隠さない・後付けしない。
🙌
6. 非エンジニア・ファースト／出力は選べる
用語をかみ砕き、操作は最小、次の一手を毎回提示。出力は Excel / HTML / PDF / PPTX から選択。

5分クイックスタート

1

フォルダに置く

キットを案件用フォルダのルートに丸ごと配置（CLAUDE.md / AGENTS.md / .cursor/ を含める）。

2

AIを起動

そのフォルダで Claude Code / Cursor / Codex を起動。入口ファイルが自動で読み込まれます。

3

情報を投げる

「この案件を見積もって」＋手元の資料を投げられるだけ投げる（RFP・画面・既存アプリ等／断片でOK）。AI が足りない分だけヒアリングします。

4

形式を選ぶ

仕上げに Excel / HTML / PDF / PPTX を指定。社内用（原価・利益率あり）と提出用は分けて出ます。

「ざっくりいくら？」程度なら 04_Quick_Estimation/（同じ原則を最小手順で）。

9ステップの流れ

案件のたびにこの順で進みます。いきなり数字を出しません（Rule 1・7）。

1
依頼受付 & ヒアリング★まず「案件情報を投げられるだけ投げて」もらう（インプットが全て）→ 足りない分だけ背景・対象OS・契約形態・予算・機密をこちらから確認 → docs/estimation_brief.md
2
案件アセスメント（Go/No-Go 判断）実現可能性・リスク・採算・前提充足度で「進めてよいか」を判断（最終判断は人間）
3
スコープ & 機能分解Must/Want/Out、フェーズ×カテゴリで展開、非機能要件チェック
4
工数積み上げ各項目に 工数(人日)×単価 と根拠。単価・係数は案件ごとに設定
5
金額構成 & 採算管理費・バッファ・税を見える化、利益率を確認（社内/提出を分離）
6
スケジュール & 体制見積明細から WBS/ガント・役割別アサインを派生
7
AIクロスチェック過大/過小・抜け漏れ・前提不足をツッコミ（人レビューの前段）
8
提案ドキュメント生成提出用見積・見積根拠資料・提案書（PPTX/Word）
9
出力 & 納品Excel/HTML/PDF/PPTX。提出前チェックリスト＋外部提出前は人間レビュー

Excelの使い方（社内用）

AI に「Excel で出力して」と頼むと、bravesoft 標準の見積もり Excel（02_Estimation_Template/estimate_template.xlsx が空テンプレ）が出ます。誰でも 3 ステップ・青いセルを変えるだけ。プログラミングの知識は要りません。

1

⚙ 設定を埋める

「設定」シートの青いセルに単価・管理費率・バッファ・税率・値引きを入力。ここを変えると全シートの金額が自動で再計算されます。

2

📋 前提条件を埋める

見積根拠・対応範囲・対象外・契約形態・納品物を記入。独立シートで、顧客説明ではまずこれを提示（「外れたら別途」の保険）。

3

✓ 見積書で明細を調整

「見積書」シートで工数・担当・項目を直し、要らない行は A 列の計上を ✓→－に。隣の見積サマリ・PM費・税込・スケジュールまで全部が連動します。

シート早わかり

シート何をする？
📖 使い方操作ガイド・色の凡例・用語（人日/準委任 等）。各シートにも「⏭ 次にやること」が書いてあります。
⚙ 設定単価表（役割別・社内/社外）・管理費率・バッファ・税率・値引き・体制維持費・更新履歴。青セル＝入力欄。
📋 前提条件見積根拠・対応範囲・対象外・契約形態・環境・納品物・検収条件（Rule 3）。独立シートで、顧客説明で最初に提示。
📊 見積サマリ案件概要＋金額サマリが1枚に（見積書の隣・数式連携）。フェーズ別の工数・金額税抜、合計税抜/消費税/税込。開いた瞬間に合計が見える（下スクロール不要）。顧客提出は A:D（採算は範囲外）。
📝 見積書顧客に渡す見積明細。計上✓で金額を出し入れ。社外金額（税抜）が主役で、社内金額・利益率・根拠・MWOは折りたたみグループ（社内用・ワンクリックで隠せる）。顧客提出は印刷範囲 A:I を PDF。
📅 スケジュール開始日・並走人数を変えるとガント・総期間が自動で動く。工数は見積書と連動。
🗣 ヒアリングシートお客様ヒアリングの観点漏れ防止チェック（51+3項目）。見積もりから分かる項目はAIが回答を事前記入。人は各観点を見て「確認」に✓を付けるだけ（✓で行が緑に色付く／上部に「確認済み○/54」）。
🛡 非機能要件チェック性能・セキュリティ・運用など29観点を、対象分野（アプリ/Web/サーバ/インフラ）ごとに 対象/対象外/要確認 で判定（Rule 2）。
🗂 案比較Must案/フル案など複数案を並べたときだけ登場。工数・金額・利益率を横並び比較。

金額は「見積サマリ」で一目、明細は「見積書」で編集——2枚は数式で連携し、明細を直すとサマリが自動更新されます。社内の原価・利益率は折りたたみグループ（列の上の [－] ボタンで隠せる）に入っていて、顧客に渡す列（見積書＝A:I／サマリ＝A:D）だけを PDF にすれば、そのまま提出物になります（原価は出ません＝Rule 10）。

🔤 優先度（MWO）列 ＝ 金額を動かすレバー

見積書のいちばん右（社内グループ内）の優先度（MWO）列が、金額調整の要です。「予算オーバー」と言われたら、値引きより先にここでスコープを調整します（Rule 5）。

M

Must（必須）

これだけで成立する最小構成。外せない機能。

W

Want（できれば）

予算調整の余地。明細では青太字。削るならまずここ。

O

Out（対象外）

やらないこと。グレー表示・0円。

★

未確定

要件が固まっていない。赤表示・0計上で一覧に残す。

明細の右に 💡 優先度（MWO）別の金額が自動集計されます。「M だけの金額」「W の合計（＝いくら削れるか）」「Must だけに絞った提示額（概算）」が一目で分かるので、① 💡 で W の合計を見る → ② W の行の計上を － にする、の順で予算に合わせられます。

安心して使うためのポイント

  • 社内数字は隠せます：社内金額・利益率は「見積書」の折りたたみグループに入っています。顧客に渡すときは列を折りたたんで印刷範囲 A:I を PDF、または AI に「提出用の HTML / PDF を生成して」と頼めば、原価・利益率は出ません（Rule 10）。
  • 数式は保護済み：金額セルは誤って上書きしないようロック（パスワード無し）。触るのは青いセルだけで OK。
  • 作り直しは AI に：行の大量追加・フェーズ追加は、AI に「見積もり Excel を再生成して」と頼むのが確実です。

仕様の詳細は 02_Estimation_Template/excel_format_spec.md、生成ツールは 05_Tools/generate_estimate_excel.py。

プロンプト定型集

AI にコピペで渡せる定型。初回プロンプトを忘れても、入口ファイル（CLAUDE.md 等）により AI が自動でキットに乗ります（Rule 14）。

① 見積もりをフルで依頼する

「案件名 / どんなプロダクトか」を見積もってください。 手元の資料（RFP・画面一覧・既存アプリのURL・API仕様・議事録など）は、断片でも全部渡します。まずそれに目を通してください。 00_AI_Instructions/workflow_guide.md に従い、資料で埋まらない分だけ（背景・対象OS・契約形態・予算感・機密区分など）をヒアリングしてください（迷う項目は「お任せ」と言います）。 そのうえで、Go/No-Go判断 → 機能分解(非機能チェック) → 工数積み上げ(根拠つき) → 金額・採算 → スケジュール → クロスチェック の順で進めてください。 金額の確定と外部提出は私の承認を取ってから進めてください。

② この案件、受けるべき？（Go/No-Go）

「案件概要」、受けるべきか判断したいです。 Step_02 に従い、実現可能性・リスク・採算の見立て・前提の充足度を評価して、 「進める / 条件付き / 見送り」の判断材料を Go/No-Go レポートにまとめてください。最終判断は私がします。

③ ざっくりいくら？（軽量・レポート不要）

ざっくりの概算を知りたいです（レポート不要、チャットでOK）。 04_Quick_Estimation の流儀で、主要機能を粗く積み上げ（PM・テスト・リリースは必ず足す）、前提と対象外を添えて、 レンジ（〜万円 / 〜人日 / 〜ヶ月）と見積精度（±%）で出してください。どんぶり勘定の一言金額はNGで。

④ 出力形式を指定する

できた見積もりを Excel / PDF / PPTX で出力してください。 output_formats.md と Step_09 に従い、社内用（原価・利益率あり）と提出用（なし）を分けて、bravesoftデザインで。 提出用は提出前チェックリストを通し、原価・利益率・個人名の混入が無いか最終確認してから出してください。

⑤ 別セッションから再開する

前回の続きをお願いします。 00_AI_Instructions/ai_context_log.md と docs/estimation_brief.md を読んで、前回までの状態を復元してから、進行中の Step を続けてください。

FAQ

相場感やどんぶり勘定で金額が出たりしませんか？

Rule 1（積み上げ厳守）で防いでいます。機能を項目まで分解し、各項目に 工数(人日)×単価 と「なぜこの工数か」の根拠を付けて積み上げます。「○○高くない？」に積み上げ根拠で即答できる状態（見積根拠資料）まで作ります。相場・記憶・一式の大きな金額は禁止です。

顧客に原価や利益率が漏れたりしませんか？

Rule 4・10 で、1つの正本から「社内用（原価・利益率あり）」と「提出用（なし）」を分けて出します。社内用HTMLには「🔒提出禁止」バナーが付き、提出前チェックリストの機密項目（原価・利益率・社内単価・個人名の混入なし）を必ず通し、人レビューを経て提出します。

「この案件、受けるべき？」だけ相談できますか？

できます（Step 02）。実現可能性・リスク・採算の見立て・前提の充足度を評価し、「進める / 条件付き / 見送り」の判断材料を Go/No-Go レポートにまとめます。最終判断は人間です（Rule 13）。

単価や管理費は案件で変えられますか？

はい。単価・管理費率・バッファ係数は案件ごとのパラメータです。estimate_master.md 冒頭の「見積もり設定」や Excel の入力セルを変えると集計が連動します。標準値は pricing_and_rates.md（暫定・要確認）。役割別・案件別に変わる前提です。

非エンジニアでも本当に使えますか？

はい（Rule 6）。AI が背景・案件概要からこちらの主導でヒアリングし、人日・準委任/請負・非機能などの用語はかみ砕いて説明します。出力形式も選べ、毎ターン「次に何をすればいいか」を提示するので迷子になりません。

抜け漏れ（PM・テスト・非機能など）は防げますか？

Rule 2 で、フェーズ×カテゴリのマトリクスで機能を逆引きし、PM・管理費／テスト／リリース・ストア申請／ドキュメントを必ず確認します。さらに nonfunctional_requirements_checklist.md を 1 回通し、性能・セキュリティ・運用などの非機能を「対象/対象外」で判断します。

ブランドの色やロゴを調整したい

配色・ロゴ・タイポは bravesoft 公式デザインシステム（ルート DESIGN.md）準拠です（primary #0066FF / ink #090A0A / navy #253A58、Noto Sans JP ＋ Roboto Mono、ロゴ assets/bravesoft-logo.png）。微調整は index.html と各テンプレ先頭の :root ブランドトークン（--brand 等）を書き換えれば全体に反映されます。

毎回ちゃんと作らないとダメ？ ざっくりだけ欲しいときは？

04_Quick_Estimation/ の軽量モードがあります。「積み上げ・前提明示・機密分離・人レビュー」の原則は守りつつ、手順だけ最小化してレンジで概算を出せます。「ちゃんとした見積もりにして」と言えば本フローに合流します。
