3. AS-IS（旧UI）と課題

旧UI（モック内の比較表示より）

  • 「稼働分析情報」画面で後からデータソースを選択
  • ソースごとに別の詳細設定画面
  • 出社滞在・従業員で2画面に分かれる
  • カラム設定は詳細設定で手入力

課題（整理）

  • 画面遷移が多く、初回セットアップの学習コストが高い
  • 列名の手入力ミスが起きやすい
  • 方式ごとのUI差分が大きく、保守・説明が重い
  • 「何をどの順で設定するか」が利用者に見えにくい

7. 改善・追加提案（モックを踏まえた次の一手）

  1. 本モックを本番UIの仕様書として固定（文言・ステップ・4方式・API）— 追加ヒアリングなしで実装着手可能。
  2. 自動提案の精度ログ（提案列 vs ユーザー修正）を取り、ルール改善に回す。
  3. 従業員ファイルと出社滞在のID突合プレビュー（一致率・不一致サンプル）を確認画面に追加。
  4. エラー時の再実行ガイド（形式不一致・必須列欠落）をステップ内に固定文言化。
  5. 多言語（JA/VI）は本提案ドキュメントと同様、UIトグルを検討（運用が日越の場合）。

8. 次アクション

  1. 本提案の内容（特に4方式・5ステップ・API）に相違がないか確認
  2. OK なら「このモックで実装方針を確定」と返信
  3. 差分があれば、修正点だけ指摘（例: 集計条件の既定値、Meraki手順）

※ 金額・工数の積み上げは依頼により行っていません。必要になった時点で別途対応します。

7. Đề xuất cải tiến thêm

  1. Chốt mockup này làm spec UI (copy, 5 bước, 4 nguồn, API) — đủ để bắt đầu implement.
  2. Log độ chính xác đề xuất cột (đề xuất vs user sửa) để tinh chỉnh rule.
  3. Preview khớp ID giữa file nhân viên và file chấm công (tỷ lệ khớp / mẫu lệch) ở bước xác nhận.
  4. Hướng dẫn lỗi cố định trong từng bước (sai format, thiếu cột bắt buộc).
  5. Đa ngôn ngữ JA/VI trên UI nếu vận hành Nhật–Việt (giống toggle tài liệu này).

8. Việc cần làm tiếp

  1. Xác nhận nội dung đề xuất (đặc biệt 4 nguồn · 5 bước · API) có đúng không
  2. OK thì trả lời “chốt mockup này làm hướng triển khai”
  3. Nếu lệch, chỉ cần chỉ ra điểm sửa (vd. mặc định giờ tổng hợp, bước Meraki)

※ Không tính chi phí/công theo yêu cầu. Khi cần sẽ làm riêng.
