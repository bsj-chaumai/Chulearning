# Step 03: スコープ & 機能分解

## 1. 目的

案件を **機能の一覧**に分解し、**Must / Want / Out of Scope** を決め、**フェーズ×カテゴリのマトリクス**で「どこに工数が要るか」を漏れなく洗い出す。あわせて**前提条件・対応範囲・対象外・契約形態・納品物**を固定し、**非機能要件チェックシート**を通す（Rule 2 / Rule 3）。
ここで作る機能一覧＝WBSの骨格が、Step 04 の工数積み上げの土台になる。

## 2. 前提条件

- Step 02 で「進める / 条件付き」の判断を得ている。
- 参照: `docs/estimation_brief.md` / `docs/go_no_go_report.md`、`03_Assessment_and_Checklists/` の `nonfunctional_requirements_checklist.md` / `assumptions_and_scope_template.md` / `deliverables_template.md` / `feature_api_mapping_template.md`（任意）。

## 3. ヒアリング項目

- 機能の優先度: 各機能は Must（必須）/ Want（あれば）/ Out（やらない）のどれか
- フェーズ分割の希望: 一括 / Phase 分割（例: Phase1 で MVP、Phase2 で拡張）/ STEP 分割
- 既存改修の場合: 流用できる部分 / 作り直す部分の線引き
- 非機能の要求水準: 性能・同時接続・可用性・セキュリティ（決済/個人情報の有無）

## 4. 作業内容

1. **機能一覧の作成**: 提供資料（RFP・画面一覧）や概要から機能を洗い出し、Must/Want/Out（要件未確定は「★未確定（別途）」）を付ける。資料が無ければ画面・ユースケースから逆算する（推測は確度を添える＝ Rule 8）。
2. **フェーズ×カテゴリのマトリクス化**（このキットの肝）:
   - **フェーズ（縦）**: 要件定義 → デザイン → 設計（内部/外部）→ 開発 → テスト/検証 → （セキュリティ）→ リリース/納品 → （運用）。横断で **PM・全体管理**。
   - **カテゴリ（横）**: 共通 / アプリ(iOS/Android) / フロント / API・サーバ / 管理画面(CMS) / インフラ / バッチ / 外部連携。
   - 各機能を「どのフェーズ×カテゴリで作業が要るか」で展開し、**PM・テスト・リリース/ストア申請・ドキュメント（納品物）が漏れていないか**を確認する。
   - API 連携/ハイブリッド案件は `feature_api_mapping_template.md` で機能↔API↔実装方式↔Phase を別表化すると、漏れ逆引きと Phase 振り分けが速い。
3. **非機能要件チェック**: `nonfunctional_requirements_checklist.md` を 1 回通し、性能 / 可用性 / セキュリティ / 運用・保守 / 拡張性 などで必要な作業を項目化、不要なら「対象外」と明記する。
4. **前提・スコープの固定**: `assumptions_and_scope_template.md`（見積根拠・対応範囲・対象外・契約形態・環境）と `deliverables_template.md`（納品物・検収条件・ソースコードの扱い）を埋める。
5. 結果を見積明細の骨格（行）として `02_Estimation_Template/estimate_master.md` に反映し、`🔔` でスコープの承認を得る。

> 出力イメージ（明細の骨格・1 行 = 1 作業項目）:
> `フェーズ | カテゴリ | 項目 | 概要 | (工数は Step 04 で) | Must/Want/Out`
