# Step 01: 依頼受付 & ヒアリング

## 1. 目的

**見積もり依頼を受けたら、いきなり数字を出さない。** まず AI の側から、**案件の背景・経緯**と**案件 / プロジェクトの概要**を含め、**何を・誰に・いつまでに・どんな前提で**見積もるのかをヒアリングして確定し、`docs/estimation_brief.md` に記録する。
ここがブレると以降すべて（工数・金額・スコープ）がブレる。**記録なしに次の Step へ進まない**（Rule 7）。

> このキットの肝の一つ。利用者に丸投げせず、**こちらから順に聞いてあげる**（選択肢式・「お任せ」可）。背景と案件概要を先に押さえると、的外れな見積もりと抜け漏れを防げる。

### 📥 最初の一手：案件情報を「投げられるだけ投げて」もらう（インプットが全て）

**見積もりの精度は、最初のインプットの質でほぼ決まる。** だから AI は、構造化ヒアリングの前に（または冒頭で）利用者に必ずこう促す:

> 「**手元にある案件の情報・資料を、できる限り全部投げてください。** きれいに整理されていなくて大丈夫です（断片・口頭メモ・スクショ・URL 何でも歓迎）。多いほど見積もりの精度が上がります。」

**投げてほしいもの（多いほど精度UP）**:
- RFP / 提案依頼 / 要件メモ / 企画書
- 画面一覧・画面イメージ・デザイン（Figma 等）
- 既存アプリ / サイト / システム（URL・スクショ・リポジトリ）
- API 仕様・DB・外部連携（決済 / SNS / ID 基盤 等）の資料
- 競合・参考サービス、議事録・チャットの抜粋
- 予算感・締切・体制の希望、過去の類似見積もり

**インプット基準（精度の高い積み上げに最低限ほしい）**: ①何を作るか（機能の概要）②対象 OS／既存システムの有無 ③予算 or 締切のどちらか ④機密区分。これが欠けると概算止まり → 足りない分だけ ❓ で補う（推測で埋めない＝ Rule 8）。

> もらった材料には**必ず目を通し**、そこから読み取れた項目は下のヒアリングから差し引く（**同じことを二度聞かない**）。「資料を見たことにして」中身を確認せずに進めない。

## 2. 前提条件

- `00_AI_Instructions/workflow_guide.md` と `ai_context_log.md` を読了している（Rule 14 のセッション開始シーケンス）。
- `docs/estimation_brief.md` が**存在しない**＝未着手。存在すれば既に Step 01 完了とみなし、続きの Step へ。
- 補助: `03_Assessment_and_Checklists/hearing_sheet.md`（ヒアリングシート）を併用すると漏れがない。見積もり Excel には同じ観点（51+3 項目）が**「ヒアリングシート」タブ**として自動添付され、見積もりから分かる項目は AI が回答を事前記入・人は「確認」に ✓ を付けるだけで観点漏れを防げる（全項目回答は不要）。

## 3. ヒアリング項目

利用者に以下を確認する。**選択肢式で聞き、「お任せ」で進められるよう推奨デフォルトを必ず添える**（Rule 6）。
AI ツールの構造化質問（AskUserQuestion 相当）を優先し、一度に詰め込みすぎない。

**まず Q1・Q2（背景・案件概要）から聞く。** ここを最初に押さえるのがこのキットの肝。

| # | 質問 | 例 / 選択肢 | 推奨デフォルト |
|---|---|---|---|
| Q1 | **案件の背景・経緯**は？（なぜ今これを？） | きっかけ / 困りごと / ビジネス目的 | （必須・要回答） |
| Q2 | **案件 / プロダクト概要**は？ | 何を作る / 対象ユーザー / 現状 / ゴール | （必須・要回答） |
| Q3 | **対象 OS・端末・チャネル**は？ | iOS / Android / Web / 管理画面 / LINE 等 | iOS + Android + 管理画面 |
| Q4 | **既存システムの有無**は？ | 新規 / 既存改修 / 一部流用 | 確認（改修なら現行資料を依頼） |
| Q5 | **契約形態**の希望は？ | 準委任 / 請負 / 上流=準委任・下流=請負 | 上流=準委任・下流=請負 |
| Q6 | **予算感**は？ | 上限あり / レンジ / 未定 | 未定（まず積み上げて提示） |
| Q7 | **締切・リリース希望日**は？ | 日付 / 急ぎ / 未定 | 未定（質優先） |
| Q8 | **出力形式**は？（**複数選択可**） | Excel / HTML / PDF / PPTX / Markdown | 社内=Excel、提出=PDF/PPTX |
| Q9 | **用途・提出先**は？ | 社内検討 / 顧客提出 / 説明会 | 社内検討 → 後で提出用 |
| Q10 | **機密区分**は？ | 社外秘 / 社内限り / 公開 | 社内限り（提出時に要確認） |
| Q11 | **渡せる資料（できる限り全部）**は？ | RFP / 要件メモ / 画面一覧 / デザイン / API 仕様 / 既存アプリ・サイト / 競合 / 議事録 | **まず全部投げてもらう**（多いほど精度UP。無ければヒアリングで補完） |
| Q12 | **制約・触れてはいけない領域**は？ | 特定技術指定 / 他社配慮 / NG事項 | なし |

> Q1・Q2 以外は「お任せ」可。お任せ回答時は推奨デフォルトを提示して同意を取り、記録する。
> **聞き方**: 一度に全部投げず、まず Q1・Q2（背景・案件概要）→ 次に Q3〜Q5（対象・既存・契約）→ 残り、の順に**こちらから誘導する**（Rule 6）。AskUserQuestion 等の選択肢式を優先。
> Q10（機密）で「社外秘」を含む場合、原価・社内単価・他案件情報を提出物や外部送信に混ぜない旨を明示する（Rule 10）。

## 4. 作業内容

0. **まず「案件情報を投げられるだけ投げてください」と促す**（インプットが全て）。資料・URL・スクショ・口頭メモ何でも可。受け取った材料に目を通し、読み取れた項目を差し引いて、足りない分だけを次でヒアリングする。
1. 上記をヒアリングする（`❓` テンプレ、構造化質問を優先）。
2. 回答を **`docs/estimation_brief.md`** に記録する（下記テンプレ。投げてもらった資料は「提供資料」に列挙）。
3. 解釈を 1〜3 行で復唱し、`🔔` 人間レビューゾーン（Rule 13）で承認を得る。
4. `ai_context_log.md` の §1「案件の基本情報」「§3 確定した見積もりの前提」を更新する。

### `docs/estimation_brief.md` テンプレート

```markdown
# 見積もりブリーフ（estimation_brief）

- 案件の背景・経緯: <Q1>
- 案件 / プロダクト概要: <Q2>
- 対象 OS・端末・チャネル: <Q3>
- 既存システム: <Q4（新規 / 改修 / 流用、現行資料の有無）>
- 契約形態（希望）: <Q5>
- 予算感: <Q6>
- 締切 / リリース希望: <Q7>
- 出力形式（複数可）: <Q8>
- 用途 / 提出先: <Q9>
- 機密区分: <Q10>
- 提供資料: <Q11（パス / URL を列挙）>
- 制約 / 注意点: <Q12>
- ブリーフ確定日: <YYYY-MM-DD>
- 承認: 利用者承認済み（YYYY-MM-DD） / 未
```

## 7. 次 Step への引き継ぎ事項

**AI → AI 引き継ぎ**: `docs/estimation_brief.md` が次 Step 以降の判断基準。特に「対象 OS」「契約形態」「予算感」「機密区分」「提供資料」を Step 02 以降で常に参照する。

**AI → 人間 次アクション**（Rule 12 / Section 11.5 適用例）:

```
✅ Step 01（ヒアリング）を完了しました
- やったこと: 案件の背景・概要・対象・契約形態・予算感・機密区分を確認し docs/estimation_brief.md に記録
- 生成ファイル: docs/estimation_brief.md

⏭ 次にあなたにやってほしいこと:
1. ブリーフの内容に相違がないか確認（特に「対象 OS」「契約形態」「予算感」）
2. OK なら「アセスメントに進んで」、修正があれば指摘

🤖 回答後に AI がやること: Step 02（案件アセスメント＝Go/No-Go 判断）に進みます
```
