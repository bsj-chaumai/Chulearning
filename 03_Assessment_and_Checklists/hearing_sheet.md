# ヒアリングシート

> Step 01 で使う。AI 側から**順に聞いてあげる**ための質問集（Rule 6/7）。選択肢式・「お任せ」可。
> 回答は `docs/estimation_brief.md` に記録する。
>
> 📎 **観点漏れ防止（Excel 版）**: 見積もり Excel（`05_Tools/generate_estimate_excel.py`）には、この観点を **「ヒアリングシート」タブ**として自動添付する（51+3 項目）。見積もりから分かる項目（オフショア利用・管理画面・多言語・脆弱性診断 等）は **AI が回答を事前記入**し、人は各観点を見て「確認」列に ✓ を付けるだけ（✓ で行が色付く／上部に確認済みカウンタ）。全項目に回答する必要はなく、**観点が抜けないこと**が目的。項目定義の正本は同スクリプトの `HEARING_ITEMS`。

## 📥 まず投げてほしい情報（多いほど精度が上がる＝インプットが全て）

ヒアリングの前に「**案件の情報・資料を、できる限り全部投げてください**」と促す（整理されていなくてOK・断片でも歓迎）。もらった材料で埋まる項目は、下のヒアリングから差し引く（二度聞かない）。

- RFP / 要件メモ / 企画書、画面一覧・デザイン（Figma）、既存アプリ・サイト（URL・スクショ）、API/DB/連携資料、競合・参考、議事録、予算・締切・体制の希望、過去の類似見積もり。
- **最低限ほしい（インプット基準）**: ①機能の概要 ②対象OS／既存の有無 ③予算 or 締切 ④機密区分。足りなければ ❓ で補う（推測で埋めない）。

## まず最初に（背景・概要）— ここが肝

- **Q1. 案件の背景・経緯**: なぜ今これを？（きっかけ / 困りごと / ビジネス目的）
- **Q2. プロダクト概要**: 何を作る？ 誰のために？ 現状とゴールは？

## 対象・前提

- **Q3. 対象 OS・端末・チャネル**: iOS / Android / Web / 管理画面 / LINE / その他
- **Q4. 既存システム**: 新規 / 既存改修 / 一部流用（改修なら現行資料は？）
- **Q5. 契約形態の希望**: 準委任 / 請負 / 上流=準委任・下流=請負
- **Q6. 連携・外部サービス**: 決済 / 地図 / SNS / ID 基盤 / 在庫 / 分析 など

## 条件

- **Q7. 予算感**: 上限あり / レンジ / 未定
- **Q8. 締切・リリース希望日**: 日付 / 急ぎ / 未定
- **Q9. 出力形式（複数可）**: Excel / HTML / PDF / PPTX / Markdown
- **Q10. 用途・提出先**: 社内検討 / 顧客提出 / 説明会
- **Q11. 機密区分**: 社外秘 / 社内限り / 公開

## 材料・制約

- **Q12. 渡せる資料**: RFP / 要件メモ / 画面一覧 / API 仕様 / 既存コード / なし
- **Q13. 制約・NG**: 特定技術指定 / 他社配慮 / 触れてはいけない領域

---

### 聞き方のコツ（Rule 6）

1. 一度に全部投げない。**Q1・Q2 → Q3〜Q6 → 残り**の順で誘導する。
2. 各質問に**推奨デフォルト**を添え、「お任せ」で進められるようにする。
3. AskUserQuestion 等の**選択肢式**を優先する。
4. 機密が「社外秘」なら、原価・社内単価・他案件を提出物/外部送信に混ぜない旨を明示（Rule 10）。
